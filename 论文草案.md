# 面向自我改进型 Agent 的循环早期预警与自适应记忆修正机制研究

## 论文草案 v0.1

---

## 摘要（草稿）

当前 LLM Agent 系统（Multi-Agent、Tool-use Agent、Memory Agent）在复杂任务执行中普遍存在无限循环、重复调用工具、Context 膨胀、错误 Memory 污染后续推理等稳定性问题。现有研究主要采用"事后检测"范式：循环行为已充分显现后才触发重规划或记忆清空。本研究将该问题重新定位为 **self-improving agent（自我改进型 Agent）在测试时（test-time）、免训练（training-free）场景下的实时自我评估机制**的一种具体实现：通过监测生成过程中的熵变化与语义漂移轨迹，在循环行为完全显现之前预测风险，并对触发漂移的具体 Memory 条目进行早期定位与轻量干预。本研究以存在性验证（是否存在可提前捕捉的预警信号）与探索性实验为主，不依赖与既有方法的正面对照实验。

---

## 1. 研究背景与动机

### 1.1 问题现象
- 无限循环（Loop）、重复调用 Tool、重复规划（Replanning）
- Context 不断膨胀，但无新增有效信息（Context Stagnation）
- 错误 Memory 污染后续推理（Error Propagation in Agent Memory）
- 明明失败却持续执行

### 1.2 核心问题的范式转移
现有工作已将"如何让 Agent 更聪明"逐步转向"如何让 Agent 更稳定、更可靠"。但目前几乎所有方法都是 **reactive（事后）** 的：等待重复行为发生多次后才判定"已循环"，随后清空上下文或整体重规划，浪费大量 token 与执行步骤。

### 1.3 与 Self-Improving Agent 的关系（叙事框架）
根据 *Self-Improving LLM Agents at Test-Time*（arXiv:2510.07841）的术语区分：

> **Self-Improving Agents**：系统在无需外部干预的情况下，通过迭代式自我精炼机制自主提升特定任务表现——agent 反复生成候选动作，用内部评分信号评估，据此调整后续步骤，不改变底层系统结构。
>
> **Self-Evolving Agents**：面向更广泛、持续性的跨环境跨任务适应，演化参数化（权重）与非参数化（记忆、工具、提示词、架构）元素，支持终身学习。

本研究的定位落在 **Self-Improving（而非 Self-Evolving）** 范畴中的 **training-free、test-time** 子类：不改模型权重，而是在单次任务执行内，依据内部信号（熵、语义漂移）实时调整记忆与执行路径，构成一种"运行时自我改进循环"。

## 2. 相关工作

### 2.1 Self-Improving / Self-Evolving Agent
| 工作 | 核心贡献 | 与本研究关系 |
|---|---|---|
| Darwin Godel Machine (ICLR 2026) | 开放式演化的自我改进 agent | 架构级参照，权重级演化，与本研究的 training-free 定位互补 |
| Gödel Agent (ACL 2025) | 自指式递归自我改进框架 | 理论参照 |
| SICA | 取消 meta/target agent 区分，agent 直接编辑自身代码库 | 说明 training-free self-modification 的可行性 |
| R-Zero (ICLR 2026) | 无需人工标注数据的自我演化推理 LLM | 数据侧参照，非本研究核心 |
| AlphaEvolve / ADAS | meta-agent 改进任务特定 agent | 工程成熟度参照 |
| A Survey of Self-Evolving Agents (arXiv:2507.21046) | 全面综述 + EvoAgentX 开源框架 + 论文列表 | 本研究 related work 的主要索引来源 |

### 2.2 循环 / 失败模式检测与分类
- **MAST（Why Do Multi-Agent LLM Systems Fail?, arXiv:2503.13657）**：标注 1600+ 条多智能体执行轨迹，提出 14 种失败模式分类，"Step Repetition" 为循环类问题的代表性类别。已开源 200+ 对话轨迹及标注。
- **Circular Reasoning（arXiv:2601.05693）**：证明纯推理模型（无工具调用）中，语义层面循环会先引发高熵反思型 token，可作为预警信号提前约 40 句话（约 1500 token）预测循环风险。**本研究的核心缺口即：该机制尚未扩展到工具调用型 Agent 场景**。

### 2.3 Memory 错误传播与修复
- **How Memory Management Impacts LLM Agents（arXiv:2505.16067）**：实证研究错误记忆在多次任务执行中的传播与放大。
- **REPAIR（arXiv:2510.01879）**：按错误率对记忆池做选择性剪枝，而非清空全部上下文（事后修复范式）。
- **TrajAD（arXiv:2602.06443）**：提出 rollback-and-retry 机制做步骤级错误定位与恢复。

### 2.4 失败轨迹标注数据集（详见第 4 节）
TRAIL、Who&When、TraceElephant、AgentPex 等，均为**事后诊断**取向，尚无面向"预测性早期干预"设计的数据集。

## 3. 研究缺口与拟解决问题

1. 现有循环检测方法几乎全部是事后检测，尚未验证工具调用场景下是否存在可提前捕捉的预警信号。
2. 现有 Memory 修复方法均为事后修复，未与预警信号结合以实现"发生前"的主动干预。
3. Self-Improving Agent 文献中的"内部评分信号"机制目前局限于任务级/输出级评估，尚未具体化到"推理循环风险"这一细粒度过程信号上。

**核心研究问题：**

> 能否在 Agent 即将进入循环状态之前，通过可观测的中间信号（生成熵、输出 embedding 相似度轨迹）提前预测风险，并定位、干预触发漂移的具体 Memory 条目，构成一种 training-free、test-time 的自我改进机制？

## 4. 确定 Benchmark（已定稿，不再继续对比选型）

### 4.1 三层结构

| 层级 | 数据集 | 规模 | 来源框架/任务 | 获取方式 | 用途 |
|---|---|---|---|---|---|
| **主数据集** | **MAST-Data** | 1,242 条标注轨迹，7 种框架（ChatDev, MetaGPT, Magentic, Open-Manus, AG2, AppWorld, HyperAgent） | 多智能体对话与工具调用 | HuggingFace: `mcemri/MAD`，文件 `MAD_full_dataset.json` | 阶段 2 核心假设验证首选：规模最大、获取门槛最低、自带 "Step Repetition" 标签，直接对口循环研究 |
| **辅助验证集** | **TRAIL** | 148 条 OpenTelemetry 轨迹，841 个 span 级错误，20+ 错误类别 | GAIA（Open Deep Research，117 条）+ SWE-bench（CodeAct） | 公开 benchmark，见 arXiv:2605.14865 | span 级颗粒度更细，用于交叉验证"循环起点前 N 步异常信号"这类精细时序分析，并测试跨任务泛化性 |
| **自建轨迹（必需，不可替代）** | 自采集 | 先做 10-20 条验证可行性，可行再扩量 | LangGraph 在 GAIA / ALFWorld 困难任务子集上运行 | 自行采集 token-level logprobs + embedding | 现有数据集均为事后标注，不含过程信号（logprobs/熵），是核心方法（早期预警）必须的原始材料 |

### 4.2 暂不使用（记录原因，避免以后反复纠结）

- **Who&When（Agents_Failure_Attribution）**、**TraceElephant**、**AgentPex**：均为合理备选，规模与场景和 MAST-Data / TRAIL 有重叠。优先把主+辅数据集跑透；若后续发现样本量或场景覆盖不足，再从这三个中按需补充。
  - Who&When：194 工具 / 183 图 / 1,448 节点，长轨迹，适合"错误传播路径"分析
  - TraceElephant：380 条轨迹（220 条失败），GitHub `TraceElephant/TraceElephant`，标注了责任 agent 与决定性失败步骤
  - AgentPex：424 条轨迹，τ²-bench 场景，GitHub `microsoft/agentpex`，规范违反类失败，可用于跨领域泛化性验证

## 5. 拟议方法框架

```
Executor Agent（执行任务，工具调用 + 生成）
      ↓ 逐步输出
Signal Extractor（提取每步的 token 级熵 + 输出 embedding）
      ↓
Early Warning Monitor（比较连续步骤的语义漂移轨迹与熵变化，
                       判断是否出现"高熵反思 + 语义收敛但无新进展"模式）
      ↓ 触发预警
Memory Localizer（回溯定位与当前漂移语义高度关联的 Memory 条目）
      ↓
Lightweight Intervention（临时屏蔽相关工具 / 插入重新聚焦提示 /
                          标记该 Memory 条目待复核，而非清空全部上下文）
      ↓
继续执行，验证是否规避了循环
```

### 5.1 与现有工作的差异化
- 现有工作（REPAIR、TrajAD）：**循环已发生 → 定位错误 → 事后回滚**
- 本研究：**预警信号出现（循环尚未完全显现）→ 定位关联 Memory → 预防性轻量干预**

## 6. 实验设计（探索性，非对照实验）

| 步骤 | 内容 |
|---|---|
| 1. 现有数据集分析 | 在 MAST-Data / TRAIL 中筛选 "Step Repetition" 及 memory 相关失败轨迹，验证"循环发生前是否存在可定位的错误注入点"这一前提假设 |
| 2. 自建轨迹采集 | 用 LangGraph/AutoGen 在 GAIA、ALFWorld 任务上运行，采集 token 熵 + embedding 轨迹 |
| 3. 信号存在性验证 | 检验循环发生前 N 步内，熵/语义漂移是否已出现可识别的异常模式（核心验证点） |
| 4. Memory 关联性验证 | 验证预警信号与具体 Memory 条目的可关联性（是否能定位到"元凶" Memory） |
| 5. 干预可行性验证 | 设计简单实时干预（如临时屏蔽工具、重新聚焦提示），观察能否将循环消灭在萌芽阶段 |
| 6. Reality Check（真实性校验） | 参考 witcheer/agentic-score-leaderboard 的 "Reality Anchor" 方法论：合成指标可能无法区分同源轨迹间的真实差异，因此需用人工标注的真实循环 vs 非循环轨迹做交叉校验，而非仅依赖自建的熵/漂移指标本身 |

**核心指标**：
- 早期检测率（Early Detection Rate）
- 提前量（领先步数 / token 数）
- 误报率（False Positive Rate）
- 干预后循环规避率
- Memory 定位准确率（人工抽样标注核验）

---

## 7. 预期贡献

1. 首次将"早期预警式循环预测"从纯推理场景扩展到工具调用型 Agent 场景。
2. 提出"预警信号 → Memory 定位 → 预防性轻量干预"的闭环机制，区别于现有事后回滚范式。
3. 将该机制纳入 self-improving agent（training-free、test-time 子类）的叙事框架，扩展该领域在"运行时可靠性"维度上的具体实现。
4. 公开自建的、含过程级信号（熵/embedding 轨迹）标注的补充数据集，弥补现有失败轨迹数据集普遍缺乏中间信号的问题。
